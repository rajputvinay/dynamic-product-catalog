# Dynamic Add Product & Search

Reference implementation for a category-driven product catalog where:

- Admin defines categories and attribute schemas.
- Add Product renders fields dynamically from category metadata.
- Product details pages adapt by category without frontend hardcoding.
- Search filters are generated by backend APIs.

## Structure

- `backend/`: Node.js REST API using built-in `http`.
- `client/`: React + Vite frontend for admin and catalog use cases.

## Run

```bash
cd backend
npm install
npm start
```

```bash
cd client
npm install
npm run dev
```

Backend runs on `http://localhost:4000`.
Frontend runs on `http://localhost:5173`.

## Backend Environment

Create `backend/.env` from `backend/.env.example` and set:

```env
PORT=4000
MONGO_URI=mongodb://127.0.0.1:27017/dynamic-catalog
```

Current sample data still runs in memory for interview simplicity, but `MONGO_URI` is now part of the backend configuration contract for the eventual MongoDB integration.

## Backend API

### Categories

- `GET /api/categories`
- `GET /api/categories/:id`
- `POST /api/categories`

### Products

- `GET /api/products?categoryId=mobile&q=nova&filters={"ram":["12 GB"]}`
- `GET /api/products/:id`
- `POST /api/products`

### Search Filters

- `GET /api/search/filters?categoryId=mobile&q=nova`

### Config

- `GET /api/config`

Returns the backend runtime config including the active `MONGO_URI` value.

## Design Notes

### Why the frontend does not change for new categories

1. Category schema lives in backend metadata.
2. Product form iterates `attributeGroups` and `contentModel`.
3. Product details iterate `detailsLayout`.
4. Search UI renders whatever filters the backend returns.

Adding a new category is a data operation, not a frontend deployment.

### Scalable schema model

- `Category`
  - identity fields
  - `attributeGroups[]`
  - `contentModel[]`
  - `detailsLayout[]`
- `Product`
  - common fields like title, SKU, brand, price
  - `attributes` map keyed by category attribute key
  - `content` map keyed by content model fields

This maps cleanly to MongoDB documents, PostgreSQL JSONB, or search index documents.

### Search / Elasticsearch alignment

This sample keeps data in memory for interview simplicity, but the API contract is ready to map to Elasticsearch/OpenSearch:

- Index one product document with normalized common fields plus `attributes.*`.
- Use category metadata to build aggregations for filterable attributes.
- Query backend first for category schema, then compose search query and aggregations centrally.
